What Is GovLens?

Understanding the purpose, the users, and the core idea — before writing a single line of code.

Think of it this way: Imagine a city has 8 departments (Public Works, Transportation, Parks, etc.). Every day, hundreds of citizens call or email to report problems — potholes, broken streetlights, building permit delays, water leaks. GovLens is the system that tracks all of these complaints, shows managers how fast they're being resolved, and uses AI to generate plain-English summaries of performance.

🏛️ The Problem It Solves

Without GovLens, city managers dig through spreadsheets to find out why permits are backlogged or why repairs take 3 weeks. GovLens puts everything on one screen — instantly.

👥 Who Uses It

Admin — full control
Manager — create & update requests, generate AI insights
Analyst — view only, read reports

📊 Core Value

Real-time visibility into SLA compliance, department performance, resolution speed, and AI-powered recommendations — all in one dashboard.

Tech Stack Explained

Every technology used, what it does, and why it's used — for complete beginners.

⚛️ React (Frontend)

What it is: A JavaScript library for building user interfaces. It lets you build the Login page, Dashboard, Requests table, and Insights tab as reusable "components".

Why React: It updates only the parts of the page that change (like updating a counter) without reloading the whole page.

☕ Spring Boot (Backend)

What it is: A Java framework that makes it easy to create REST APIs. When the frontend says "give me all requests", Spring Boot is the part that handles that request, talks to the database, and sends back the data.

Why Spring Boot: Industry standard for Java backends. Built-in security, JPA for database, REST support out of the box.

🐘 PostgreSQL (Database)

What it is: A powerful open-source relational database. Stores all users, departments, service requests, and AI insight caches in structured tables.

Why PostgreSQL: Handles large datasets well (10,000+ requests), supports complex queries for calculating SLA compliance, averages, and counts.

🤖 OpenAI GPT-4 (AI)

What it is: An AI language model accessed via API. You send it the department's metrics as a prompt, and it returns a paragraph of analysis in plain English.

Fallback: If no API key is set, GovLens uses a rule-based engine (if/else logic) to generate insights from the numbers.

🔐 JWT (Authentication)

What it is: JSON Web Token. After login, the backend sends a token (a long encoded string). The frontend stores this and sends it with every request.

Why JWT: Stateless — the server doesn't need to remember who's logged in. The token itself contains the username and role.

🔗 JPA / Hibernate (ORM)

What it is: A tool that maps Java classes to database tables automatically. Instead of writing raw SQL, you write Java code and Hibernate translates it to SQL.

Example: requestRepo.findAll() automatically runs SELECT * FROM service_requests

Complete Login Flow — Step by Step

1

User types username + password and clicks Sign In

React captures the form values in state: const [username, setUsername] = useState(''). When Sign In is clicked, it calls a handleLogin() function.

2

React sends a POST request to the backend

The frontend calls POST /api/auth/login with body {"username": "admin", "password": "admin123"}

3

Spring Boot checks the credentials against PostgreSQL

It runs a query to find the user by username, then uses BCrypt to verify the password. BCrypt is a hashing algorithm — passwords are never stored as plain text.

4

If valid, Spring Boot generates and returns a JWT token

The token contains the username and role (e.g. "MANAGER"). It's returned as JSON: {"token": "eyJ...", "role": "MANAGER", "username": "manager1"}

5

React stores the token and redirects to the dashboard

Token is saved in localStorage. The role is saved in React Context so every component knows what the current user can see/do. User is redirected to the Overview tab.

6

Every future API call includes the token in the header

All subsequent requests include: Authorization: Bearer eyJhbGc.... Spring Boot's JwtFilter reads this token, verifies it, and extracts the role for permission checks.

React — handleLogin function 
const handleLogin = async () => {
  const response = await fetch('/api/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username, password })
  });

  if (response.ok) {
    const data = await response.json();
    localStorage.setItem('token', data.token);   // save JWT
    localStorage.setItem('role', data.role);     // save role
    navigate('/dashboard');                     // go to dashboard
  } else {
    setError('Invalid username or password');
  }
};
Spring Boot — AuthController.java 
@PostMapping("/api/auth/login")
public ResponseEntity<?> login(@RequestBody LoginRequest req) {

  // 1. Find user in database
  User user = userRepo.findByUsername(req.getUsername())
      .orElseThrow(() -> new BadCredentialsException("User not found"));

  // 2. Check password using BCrypt
  if (!passwordEncoder.matches(req.getPassword(), user.getPasswordHash())) {
      return ResponseEntity.status(401).body("Invalid credentials");
  }

  // 3. Generate JWT token with username + role inside it
  String token = jwtUtil.generateToken(user.getUsername(), user.getRole());

  // 4. Return token + role to frontend
  return ResponseEntity.ok(Map.of(
      "token", token,
      "role", user.getRole(),
      "username", user.getUsername()
  ));
}

The 5 Request Statuses Explained

● OPEN — Just submitted, not yet assigned to anyone.

● IN PROGRESS — Someone is actively working on it.

● RESOLVED — Work is done, citizen issue is fixed. Resolution timestamp is recorded here — this feeds into Avg Resolution Time.

● CLOSED — Officially closed out and archived.

● ESCALATED — Flagged as urgent — either too old or a citizen complained it wasn't fixed.

🔢 Priority Score Formula

Calculated by a scheduled job running every hour in Spring Boot (@Scheduled):

// Higher score = needs attention sooner
ageScore = hoursSinceSubmit × 0.5
slaScore = breachingIn24h ? 50 : 0
priorityScore = (6 - priorityLevel) × 10
// P1=50pts, P2=40pts, P3=30pts, P4=20, P5=10

totalScore = ageScore + slaScore + priorityScore

Spring Boot — ServiceRequest.java (JPA Entity with SLA auto-calculation) 
@Entity
@Table(name = "service_requests")
public class ServiceRequest {

    @Id @GeneratedValue
    private Long id;

    private String title, description, category, status, assignedTo;

    @ManyToOne
    private Department department;

    private String priority;         // "P1" through "P5"
    private LocalDateTime submittedAt;
    private LocalDateTime resolvedAt;
    private LocalDateTime slaDeadline; // auto-calculated below
    private Double priorityScore;

    // This runs automatically BEFORE the row is saved to the database
    @PrePersist
    public void prePersist() {
        this.submittedAt = LocalDateTime.now();
        this.status = "OPEN";
        // Set SLA deadline based on priority
        this.slaDeadline = switch(priority) {
            case "P1" -> submittedAt.plusHours(4);
            case "P2" -> submittedAt.plusHours(12);
            case "P3" -> submittedAt.plusHours(48);
            case "P4" -> submittedAt.plusDays(5);
            default  -> submittedAt.plusDays(10); // P5
        };
    }
}

The 3-Layer AI System

1

Layer 1 — Cache Check (CACHED badge)

First, the system checks if there's already an insight for this department generated less than 6 hours ago. If yes — return it instantly. No API call needed. This prevents burning money on OpenAI for every page reload.

2

Layer 2 — OpenAI GPT-4 (AI GENERATED badge)

If the cache is expired (older than 6 hours), the backend builds a prompt with the department's real metrics and sends it to OpenAI. The response is saved to the ai_insights table for future caching.

3

Layer 3 — Rule-Based Fallback (RULE-BASED badge)

If no OpenAI API key is configured (as in the demo setup), a built-in if/else engine generates a useful insight based on the numbers alone. No AI needed — just logic.

Spring Boot — AIInsightService.java (complete logic) 
public AIInsight getInsight(Long departmentId) {

    // LAYER 1: Check cache — is there a fresh insight (under 6 hours)?
    Optional<AIInsight> cached = insightRepo
        .findByDeptIdAndGeneratedAtAfter(departmentId, LocalDateTime.now().minusHours(6));
    if (cached.isPresent()) {
        cached.get().setSource("CACHED");
        return cached.get();
    }

    // LAYER 2: Try OpenAI if API key is configured
    if (openAiKeyConfigured) {
        DeptMetrics metrics = deptService.getMetrics(departmentId);
        String prompt = buildPrompt(metrics);  // builds the message to send
        String aiText = openAiClient.complete(prompt);
        AIInsight insight = new AIInsight(departmentId, aiText, "AI_GENERATED");
        return insightRepo.save(insight);
    }

    // LAYER 3: Fall back to rule-based engine
    return generateRuleBased(departmentId);
}

// Rule-based fallback — pure if/else logic
private AIInsight generateRuleBased(Long deptId) {
    DeptMetrics m = deptService.getMetrics(deptId);
    String text;

    if (m.getSlaCompliance() < 70) {
        text = m.getName() + " is critically underperforming with "
             + m.getSlaCompliance() + "% SLA compliance. Immediate action required.";
    } else if (m.getAvgResolutionHours() > 72) {
        text = m.getName() + " resolution times exceed targets. Consider staff reallocation.";
    } else {
        text = m.getName() + " is performing within acceptable parameters.";
    }
    return new AIInsight(deptId, text, "RULE_BASED");
}

Authentication & JWT

How login works, what a JWT token is, and how Spring Boot verifies every request.

🔑 What Is a JWT Token?

A JWT (JSON Web Token) is a long encoded string with 3 parts separated by dots:

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJtYW5hZ2VyMSIsInJvbGUiOiJNQU5BR0VSIn0.abc123

Part 1: Header (algorithm). Part 2: Payload (username + role). Part 3: Signature (prevents tampering). The server can verify it without storing anything — the signature proves it was issued by your server.

🛡️ Spring Security Filter

Every incoming request passes through JwtFilter.java before reaching any controller. It:

1. Reads the Authorization header
2. Extracts and verifies the JWT
3. Loads the username and role
4. Sets the security context (Spring now knows who you are)
5. Passes the request to the controller

Spring Boot — JwtFilter.java (intercepts every request) 
@Component
public class JwtFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req, ...) {

        // 1. Get the Authorization header
        String authHeader = req.getHeader("Authorization");

        // 2. Check it starts with "Bearer "
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(req, res); // no token, continue (will get 403 later)
            return;
        }

        // 3. Extract the token part (after "Bearer ")
        String token = authHeader.substring(7);

        // 4. Verify token + extract username and role
        String username = jwtUtil.extractUsername(token);
        String role = jwtUtil.extractRole(token);

        // 5. Set authentication in Spring Security context
        UsernamePasswordAuthenticationToken auth =
            new UsernamePasswordAuthenticationToken(username, null, List.of(new SimpleGrantedAuthority(role)));
        SecurityContextHolder.getContext().setAuthentication(auth);

        filterChain.doFilter(req, res); // proceed to the actual controller
    }
}

The SLA System

What SLA means, how deadlines are calculated, and how compliance percentage is tracked.

SLA = Service Level Agreement — it's the deadline by which a request MUST be resolved. Different priorities have different deadlines. Missing the deadline means the request is counted as "SLA breached" and lowers your compliance percentage.
PriorityDeadlineMeaningExample Use Case
P1 Critical4 hoursLife/safety at risk — must resolve same dayGas leak, fire hazard, collapsed road
P2 High12 hoursMajor disruption — same day resolutionTraffic signal down, sewage overflow
P3 Medium48 hoursNormal service — resolve within 2 daysPothole, streetlight out, graffiti
P4 Low5 daysMinor issue — resolve within a weekPark bench broken, missing street sign
P5 Minimal10 daysCosmetic — resolve within 2 weeksLandscaping request, noise complaint

📐 SLA Compliance Formula

SLA Compliance % = (Requests resolved BEFORE deadline ÷ Total resolved requests) × 100

Currently showing 71.4% — below the 80% target. This means out of every 100 requests that were resolved, only 71 were resolved before their deadline. 29 took too long.

Seed Data

Why the Requests tab shows 0 records, how to fix it, and how to make the data look realistic.

Why Requests Tab Shows 0 Records: The backend API is returning an empty array because no data has been inserted into the service_requests table yet. You need to seed the database with 10,000 realistic requests.
Spring Boot — DataSeeder.java (runs on startup if DB is empty) 
@Component
public class DataSeeder implements ApplicationRunner {

    @Autowired ServiceRequestRepository requestRepo;
    @Autowired DepartmentRepository deptRepo;

    private final Random random = new Random();

    String[] titles = {
        "Pothole repair needed", "Street light outage", "Building permit review",
        "Water main leak", "Park equipment broken", "Road line fading",
        "Sewage backup", "Graffiti removal", "Noise complaint"
    };
    String[] categories = {"Infrastructure", "Permits", "Utilities",
                           "Maintenance", "Safety", "Complaint"};
    String[] statuses = {"OPEN", "IN_PROGRESS", "RESOLVED", "CLOSED", "ESCALATED"};
    String[] priorities = {"P1", "P2", "P3", "P4", "P5"};

    @Override
    public void run(ApplicationArguments args) {
        if (requestRepo.count() > 0) return; // skip if already seeded

        List<Department> depts = deptRepo.findAll();
        List<ServiceRequest> batch = new ArrayList<>();

        for (int i = 0; i < 10000; i++) {
            ServiceRequest sr = new ServiceRequest();
            sr.setTitle(titles[random.nextInt(titles.length)]);
            sr.setCategory(categories[random.nextInt(categories.length)]);
            sr.setDepartment(depts.get(random.nextInt(depts.size())));
            sr.setPriority(priorities[random.nextInt(priorities.length)]);

            // Spread submissions over past 12 months
            int daysAgo = random.nextInt(365);
            sr.setSubmittedAt(LocalDateTime.now().minusDays(daysAgo));

            // 50% chance it's resolved
            String status = statuses[random.nextInt(statuses.length)];
            sr.setStatus(status);
            if (status.equals("RESOLVED") || status.equals("CLOSED")) {
                sr.setResolvedAt(sr.getSubmittedAt().plusHours(random.nextInt(72)));
            }

            batch.add(sr);
            if (batch.size() == 500) {       // insert in batches of 500
                requestRepo.saveAll(batch);
                batch.clear();
            }
        }
        if (!batch.isEmpty()) requestRepo.saveAll(batch);
        System.out.println("✅ Seeded 10,000 service requests");
    }
}

Live Data Strategy

Should you use dummy data or real data? How to get real government data. What employers actually want to see.

The honest answer: Dummy/seeded data is completely normal and expected for a government analytics portfolio project. No city will give you their real database. Real data has citizens' personal information (PII) which creates legal issues. Employers don't care if data is real — they care if the system is architecturally sound.

Option A — Smart Seed Data (Recommended for Portfolio)

Make your fake data statistically realistic. Employers are impressed by this.

• Parks & Recreation peaks in summer (Jun-Aug)
• Building permits are consistent year-round
• P1 requests are rare — only 3-5% of total
• ~71% resolve before SLA (matching your current 71.4% shown)

Current Bugs & Fixes

Every known issue in the current state of the project and exactly how to fix each one.

🔴 Bug 1 — Requests Tab Shows "0 total records"

Cause: No data in the service_requests table.
Fix: Add the DataSeeder.java class from Section 13. It will auto-run when Spring Boot starts and insert 10,000 requests if the table is empty.
Verify: Check in PostgreSQL: SELECT COUNT(*) FROM service_requests; should return 10000.

🔴 Bug 2 — AI Insights Shows "Failed to load insights"

Cause: The GET /api/insights endpoint is throwing an error — likely the ai_insights table doesn't exist, or there's a NullPointerException in the service.
Fix Step 1: Run CREATE TABLE ai_insights (...) from Section 8.
Fix Step 2: In your React fetch, add error handling: .catch(err => setError("Failed to load insights.")). Also check that the endpoint returns an empty array [] (not null) when the table has no rows.

🟡 Bug 3 — No "Generate Insight" Controls Visible

Cause: The dropdown and Generate button on the AI Insights page are either not rendered or the role check is failing.
Fix: Make sure you're logged in as manager1 or admin. ANALYST accounts cannot see the generate controls. Check that role is stored in React Context after login and the conditional rendering logic is {role === 'MANAGER' || role === 'ADMIN'}.

🟡 Bug 4 — Priority Score Not Updating

Cause: The scheduled job to recalculate priority scores hasn't been implemented yet.
Fix: Add a @Scheduled(cron = "0 0 * * * *") method in a service class that runs every hour, loops through all OPEN/IN_PROGRESS requests, and updates their priority_score using the formula from Section 6.
